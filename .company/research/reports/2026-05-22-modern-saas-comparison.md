## 調査対象サービスと主なUI/UXの特徴

### 1. Linear (linear.app)

- **URL**: https://linear.app
- **カテゴリ**: プロジェクト管理 / イシュートラッカー

**特徴**

1. **LCHカラーシステム＋カスタムテーマ**: HSLではなくLCH色空間でカラーパレットを設計。ベースカラー・アクセントカラー・コントラスト値の3変数でテーマ全体を生成するため、ダークモードでも色の知覚的明度が均一に保たれる。デフォルトのダーク背景は `#0A0A0B` 付近のウォームグレー。
2. **徹底したキーボードファースト設計**: `G` → `I`（Issuesへ移動）、`C`（新規イシュー作成）など、ほぼすべての操作をキーボードショートカットだけで完結できる。コマンドパレット（`⌘K`）がナビゲーションの中心。
3. **情報密度と余白のバランス**: リスト行の高さを最小化しつつ、左端のステータスアイコン・優先度インジケーター・担当者アバターを行内に収める。スキャン速度を優先した「高密度だが圧迫感のない」レイアウト。
4. **Inter Displayフォント活用**: 見出しにInter Displayを採用し、通常テキストはInter。フォントフィーチャー設定（`cv01`等）を有効化してブランド個性を出す。
5. **サブピクセル精度のモーション**: AnimatePresenceとspring物理ベースのアニメーションで、パネル展開やドラッグ＆ドロップが「吸い付く」感触。フレーム単位でずれない。

**このアプリに取り入れたいポイント**

- コマンドパレット（`⌘K`）によるイベント検索・画面遷移の高速化
- LCHベースのカラートークン設計（既存のCSS変数を整理・拡張）
- サイドバーの「選択中アイテム」インジケーターをスプリングアニメーションに変更
- ショートカットヒント表示（`kbd` タグのスタイルは既に実装済みなので表示箇所拡充）

### 2. Craft (craft.do)

- **URL**: https://craft.do
- **カテゴリ**: ドキュメント作成 / ノート

**特徴**

1. **タイポグラフィ主導のレイアウト**: 本文・見出しの文字サイズ比率、行間、文字間隔がすべて黄金比・Modular Scaleに基づいて設計。コンテンツ自体が主役になる余白設計。
2. **App Style（ドキュメント色を全UIに反映）**: ドキュメントのカラーテーマがサイドバー・ツールバーにも波及し、UIクロム全体が統一された雰囲気を持つ。
3. **スムーズなブロック操作**: ドラッグ＆ドロップによるブロック並び替えが60fpsで動き、掴んだブロックが浮き上がる影付きフィードバックを伴う。
4. **ネイティブ品質のトランジション**: macOS Native AppライクなTransition — パネルのスライドイン/アウト、ページ遷移のクロスフェードが OS アニメーション設定（`prefers-reduced-motion`）に従う。
5. **読みやすさ最優先の最大幅制限**: 本文エリアを `max-width: 700px` 前後に固定し、ワイドスクリーンでも文字が長くなりすぎない。余白は「行の外側」に積み増す設計。

**このアプリに取り入れたいポイント**

- イベント詳細モーダルの本文テキストを `leading-relaxed`・`max-w-prose` で制限し、読みやすさを向上
- モーダルのタブ（詳細/写真）を Craft のパネル切替のようなスライドアニメーションに変更
- タイポグラフィのModular Scale整理（現状は `text-xs`/`text-sm`/`text-base` が混在）

### 3. Vercel Dashboard (vercel.com/dashboard)

- **URL**: https://vercel.com/dashboard
- **カテゴリ**: デプロイ管理 / 開発者向けダッシュボード

**特徴**

1. **「コンテンツ＝カード」の情報設計**: プロジェクトカード1枚にデプロイ状態・ドメイン・最終更新日を収め、一覧スキャンで全体を把握できる。必要以上の情報は表示しない「制約による明快さ」。
2. **モノクロームグレースケール基調**: UI全体をほぼグレースケール（白〜黒）で構成し、ステータスカラー（緑＝成功、赤＝失敗）のみを彩色することで異常を即座に視認できる。
3. **サイドバーリデザイン（2026年2月展開）**: 従来のトップナビゲーションからサイドバーナビに移行。折りたたみ可能でコンテンツを最大化できる。
4. **トースト通知の最小化**: Syncing/エラーの状態通知を画面右下の小さなトーストに限定。メインコンテンツを邪魔しない。
5. **デベロッパーセントリックなUX文言**: 「Deploying…」「Ready」「Error」など、状態をワンワードで表現するUI文言の統一。

**このアプリに取り入れたいポイント**

- 現状の「Cloud Syncing...」トーストは既にVercel的UXを取り入れているが、エラートースト右下固定位置を統一
- イベントカードの情報密度整理：会場名・日程・地域だけの3点セットに絞る
- 準備物リスト画面のステータス列を「緑＝完了、赤＝未着、グレー＝未設定」の信号機ロジックに統一

### 4. Cron (cron.com)

- **URL**: https://cron.com
- **カテゴリ**: カレンダー / スケジューリング

**特徴**

1. **ダークテーマ＋カラーコーディングされたカレンダー**: 複数カレンダーを色分けで管理。背景は深いダークグレーで、イベントチップのカラーが映える。
2. **キーボードショートカット中心の操作**: `T`（今日に戻る）、`N`（新規イベント）、`1/2/3`（日/週/月切替）など、Googleカレンダー互換のショートカット体系。
3. **右サイドパネルのコンテキスト表示**: イベントをクリックすると画面右にスライドインするパネルで詳細を編集。メイン画面から離れずに操作が完結する。
4. **フォーカスブロック機能**: 「作業中」「会議不可」などのブロックをワンクリックで設定できる特殊イベントタイプ。
5. **コンパクトかつ情報密度の高いカレンダーセル**: 週ビューのセル内に時刻・タイトル・参加者アバターを収めながら、クリック可能な最小タップターゲットを確保。

**このアプリに取り入れたいポイント**

- 週ビューのセルデザイン（現状の月グリッドは日別イベントが圧縮されすぎる）
- カレンダー上のイベントチップに「準備進捗バー」を薄く重ねる（既にprepProgressMapは実装済み）
- モバイル週ビューの横スワイプ週送りナビゲーション

### 5. Notion (notion.so)

- **URL**: https://notion.so
- **カテゴリ**: オールインワンワークスペース / データベース

**特徴**

1. **マルチビュー切替（テーブル/カレンダー/ボード/ギャラリー）**: 同一データベースを複数のビューで表示。カレンダービューではDate型プロパティを軸に月・週表示を切替可能。
2. **インラインデータベースのミニマル化（2025年UI更新）**: ビュータブが非表示になりよりクリーンな外観に。スライダーアイコンに設定が集約された。
3. **ブロックエディタの柔軟性**: `/` コマンドで任意のブロックを挿入。コンテンツ構造をユーザーが自由に決められるため、テンプレートの再利用性が高い。
4. **右パネル非表示オプション（Notion Calendar）**: カレンダー表示時に右サイドパネルを折りたたむことでカレンダーエリアを最大化できる。
5. **プロパティの柔軟な型定義**: テキスト・数値・日付・セレクト・マルチセレクト・ファイル・ユーザーなど多様な型。フォーム的な編集UIも統一されている。

**このアプリに取り入れたいポイント**

- イベント詳細モーダルのフィールドをNotionの「プロパティ行」スタイルに整理（ラベル左、値右）
- 準備物リストに「テンプレート読み込み」機能を追加（Notionのページテンプレートに相当）
- カレンダービューの「+」から直接イベント作成する際のインラインフォーム（現状はモーダルが開く）

### 6. Fathom Analytics (usefathom.com)

- **URL**: https://usefathom.com
- **カテゴリ**: プライバシーファースト Web アナリティクス

**特徴**

1. **シングルページダッシュボード**: ユニーク訪問者・PV・直帰率・平均滞在時間をスクロールなしで1画面に収める。タブや深いナビゲーションを持たない。
2. **数値の大きさでメトリックを強調**: KPIの数字を大きなフォントサイズで表示し、ラベルは小さく。視線が自然に重要指標へ誘導される。
3. **ライト/ダーク自動切替**: OS設定の `prefers-color-scheme` を参照し、手動オーバーライドも可能。両テーマでUIの情報量に差がない。
4. **1秒以下のページロード**: 軽量実装を徹底。アニメーションは必要最小限。
5. **フィルター・日付範囲の即座反映**: ページリロードなしでグラフと数値が更新される。ローディングスケルトンを使わず、値がフェードインする。

**このアプリに取り入れたいポイント**

- サイドバーのイベント統計（total件数など）をFathomライクな「大きい数字＋小さいラベル」スタイルに
- 準備物リスト画面の進捗率表示をシングル大数字（例：「68%完了」）で見せるサマリーブロック追加
- ダッシュボード的なカレンダー上部のKPIバー（今月イベント数・準備完了率・予算合計）

---

### 7. Raycast (raycast.com)

- **URL**: https://raycast.com
- **カテゴリ**: ランチャー / コマンドパレット型アプリ

**特徴**

1. **コマンドパレットUIがプロダクト自体**: 検索バー1本から全機能にアクセスする思想。UIの複雑さを「検索で解決」に還元。
2. **モノクロームダーク＋アクセントグラデーション**: `#07080A` 系の深いダーク背景に、赤〜ピンクの斜めグラデーションラインをアクセントとして使用。カテゴリ別のカラーアクセントはアイコンのみに限定。
3. **Inter + フォントフィーチャー**: `font-feature-settings: "calt", "kern", "liga", "ss03"` でInter文字の代替グリフを有効化しブランド個性を演出。
4. **6〜10pxの小さな角丸＋1px hairlineボーダー**: カードの角丸をあえて小さくし、シャープで精密な印象を与える。ボーダーはほぼ不透明度10%以下の白。
5. **スプリングアニメーションによるパレット表示**: コマンドパレットが0.15秒程度のスプリングで現れる。遅延ゼロで即座に反応する体感速度を実現。

**このアプリに取り入れたいポイント**

- `⌘K` コマンドパレット（イベント検索・ビュー切替・新規作成を統合）の実装
- パレットUIのデザイントークン参考：ダーク背景・1pxボーダー・8pxの角丸
- キーボードショートカット一覧モーダル（`?` キーで表示）

## 現状アプリとの比較

| 観点 | 現状 | モダンSaaS標準 | ギャップ |
|------|------|--------------|---------|
| **タイポグラフィ** | Inter + Noto Sans JP、サイズが `text-xs` 〜 `text-base` に散在 | Modular Scaleに基づく階層（4〜5段階）、Inter Displayを見出しに使用 | スケールの体系化不足。見出し・本文・キャプションの差が小さい |
| **余白・グリッド** | `p-4`〜`p-8` の混在、モバイルは `p-4 pb-20` で底部が大きい | 8px基準グリッド、セクション間余白は一定の倍数で統一 | 余白値の統一基準がなく、コンポーネント毎に異なる |
| **カラーシステム** | CSS変数7本（`--primary` `--accent` 等）、ダーク変数あり | LCHベースのトークン体系、セマンティックカラー（success/warning/danger）が明示 | セマンティックカラーが未定義（エラー赤・成功緑のトークンがない） |
| **モーション・アニメーション** | motion/reactのAnimatePresence、フェードイン中心 | スプリング物理ベース、トランジション時間 120〜250ms、`prefers-reduced-motion` 対応 | アニメーション種類がほぼフェードのみ。スプリングやスライドが少ない |
| **コンポーネントの一貫性** | ボタン・カード・バッジのスタイルが個別に定義 | デザイントークンから生成されたコンポーネントライブラリ、Storybook等で管理 | コンポーネント定義がApp.tsxに直書きされているケースが多い |
| **モバイルUX** | 4タブ切替（一覧/月/週/日）、ボトムナビ未実装、タップターゲットがまばら | 固定ボトムナビゲーション（44px以上）、スワイプジェスチャー、ハプティクス想定 | モバイルナビの回遊導線が弱い。ボトムナビなしで全機能にアクセスできない |
| **情報密度** | カレンダーセルにイベントチップ（最大2件＋+N件）、ホバーカードあり | セル内進捗バー、優先度インジケーター、担当者アバター | 準備進捗のprepProgressMapがカレンダーチップに未反映 |

## 実装提案（優先度順）

### 🔴 高優先度（5月末までに）

#### 1. モバイルボトムナビゲーションバー
- **参考**: Cron、Fathom Analytics
- **現状の問題**: モバイルでは4タブ切替がカレンダー上部に存在し、メインコンテンツが圧迫される。ボトムナビが存在しないため、ビュー切替のタップ範囲が狭い。
- **実装内容**:
  ```jsx
  // 画面下部に固定するボトムナビバー（モバイル専用）
  <nav className="fixed bottom-0 inset-x-0 h-16 bg-white/95 backdrop-blur-md border-t border-slate-100 flex items-center justify-around z-40 lg:hidden safe-area-inset-bottom">
    {tabs.map(tab => (
      <button key={tab.id} className="flex flex-col items-center gap-1 min-w-[56px] py-2">
        {tab.icon}
        <span className="text-[10px] font-bold">{tab.label}</span>
      </button>
    ))}
  </nav>
  ```
- **実装後の変化**: モバイルのビュー切替が片手親指圏内に収まり、操作性が大幅に向上。カレンダーの表示エリアが増加。

#### 2. コマンドパレット（⌘K）
- **参考**: Linear、Raycast
- **現状の問題**: 検索バーは存在するが、会場・クライアント名の絞り込みのみ。ビュー切替や新規イベント作成はマウス操作必須。
- **実装内容**:
  ```tsx
  // ⌘K / Ctrl+K でトグルするコマンドパレット
  // 候補: イベント検索結果、ビュー切替コマンド、新規イベント作成
  <div className="fixed inset-0 bg-black/40 backdrop-blur-sm z-50 flex items-start justify-center pt-[20vh]">
    <div className="w-full max-w-xl bg-white dark:bg-zinc-900 rounded-2xl shadow-2xl border border-slate-100 dark:border-white/10 overflow-hidden">
      <input className="w-full px-5 py-4 text-base outline-none bg-transparent" placeholder="イベント検索、コマンド入力..." />
      <div className="border-t border-slate-100 max-h-80 overflow-y-auto">
        {/* 候補リスト */}
      </div>
    </div>
  </div>
  ```
- **実装後の変化**: PC利用の編集者3名の操作効率が飛躍的に向上。マウスなしで全機能にアクセス可能になる。

#### 3. カレンダーセルへの準備進捗バー表示
- **参考**: Cron、Linear
- **現状の問題**: `prepProgressMap` はホバーカードにのみ表示され、カレンダー上のイベントチップからは準備状況が不明。
- **実装内容**:
  ```tsx
  // EventChipの下部に薄い進捗バーを追加
  const progress = prepProgressMap[ev.id];
  const pct = progress?.total ? (progress.done / progress.total) * 100 : null;
  // ...
  {pct !== null && (
    <div className="absolute bottom-0 inset-x-0 h-0.5 rounded-full bg-black/10">
      <div className="h-full bg-emerald-400 rounded-full transition-all" style={{ width: `${pct}%` }} />
    </div>
  )}
  ```
- **実装後の変化**: カレンダーを一覧するだけで「どのイベントの準備が遅れているか」を視覚的に把握できる。

#### 4. セマンティックカラートークンの整備
- **参考**: Vercel Dashboard、Linear
- **現状の問題**: 成功・警告・エラー・情報の色が各コンポーネントに直書きされており、テーマ変更時に漏れが生じる。
- **実装内容**（`src/index.css` の `@theme` ブロックに追記）:
  ```css
  --color-success: #10b981;
  --color-success-bg: #ecfdf5;
  --color-warning: #f59e0b;
  --color-warning-bg: #fffbeb;
  --color-danger: #ef4444;
  --color-danger-bg: #fef2f2;
  --color-info: #3b82f6;
  --color-info-bg: #eff6ff;
  ```
- **実装後の変化**: 準備物リストのステータス表示・エラートースト・バッジの色が全コンポーネントで一貫し、ダークモードへの対応も一元化できる。

### 🟡 中優先度（6月）

#### 5. タイポグラフィ Modular Scale の整理
- **参考**: Craft.do
- **現状の問題**: `text-xs`（12px）と `text-sm`（14px）が混在。見出しと本文の差が小さく、視覚的ヒエラルキーが弱い。
- **実装内容**:
  ```css
  /* @theme に追加 */
  --text-display: 2rem;      /* 32px: ページタイトル（TODAYの日付など） */
  --text-heading: 1.125rem;  /* 18px: セクション見出し */
  --text-body: 0.875rem;     /* 14px: 本文 */
  --text-caption: 0.75rem;   /* 12px: ラベル・メタ情報 */
  --text-micro: 0.625rem;    /* 10px: バッジ・タグ */
  ```
- **実装後の変化**: 画面内の文字要素に明確な役割分担が生まれ、情報の優先度を視線で読み取れるようになる。

#### 6. スプリングアニメーションの導入
- **参考**: Linear、Raycast
- **現状の問題**: モーションがほぼ `opacity` フェードのみ。モーダル開閉やサイドバートグルに物理的な重みがない。
- **実装内容**:
  ```tsx
  // モーダル開閉をスプリングに変更
  <motion.div
    initial={{ opacity: 0, scale: 0.96, y: 8 }}
    animate={{ opacity: 1, scale: 1, y: 0 }}
    exit={{ opacity: 0, scale: 0.96, y: 8 }}
    transition={{ type: "spring", stiffness: 400, damping: 30 }}
  >
  ```
- **実装後の変化**: UIの質感が「Webアプリ感」から「ネイティブアプリ感」に近づく。開閉操作の体感速度が向上。

#### 7. モバイルでの横スワイプ週送り
- **参考**: Cron
- **現状の問題**: モバイル週ビューで前後の週に移動するには「<」「>」ボタンを押す必要があり、片手操作が難しい。
- **実装内容**: `react-swipeable` または `motion` のドラッグジェスチャーでスワイプ検知し、`setMobileWeekRowIndex` を呼び出す。
- **実装後の変化**: モバイルの週ビュー操作が直感的になり、現場スタッフのスケジュール確認がスムーズになる。

#### 8. サイドバー統計のKPI表示スタイル改善
- **参考**: Fathom Analytics
- **現状の問題**: サイドバーの「すべてのイベント: N件」がリストアイテムとして表示されており、数値の視覚的な重みが弱い。
- **実装内容**:
  ```tsx
  // TODAY セクションの下に KPI カードを追加
  <div className="grid grid-cols-2 gap-2">
    <div className="bg-slate-50 rounded-xl p-3">
      <div className="text-2xl font-black text-slate-800">{stats.total}</div>
      <div className="text-[10px] font-bold text-slate-400 uppercase tracking-wider">総イベント</div>
    </div>
    <div className="bg-emerald-50 rounded-xl p-3">
      <div className="text-2xl font-black text-emerald-700">{preparedPct}%</div>
      <div className="text-[10px] font-bold text-emerald-500 uppercase tracking-wider">準備完了</div>
    </div>
  </div>
  ```
- **実装後の変化**: サイドバーを開くだけでイベントの全体状況を数字で把握できる。

#### 9. イベント詳細モーダルのプロパティ行スタイル統一
- **参考**: Notion
- **現状の問題**: モーダル内のフォームフィールドが縦積みのinputスタイルで、情報の一覧性が低い。
- **実装内容**: 各フィールドを「ラベル（左、w-24固定）＋値（右、flex-1）」のhorizontal行に並べ、Notionのプロパティ行UIに近づける。
- **実装後の変化**: モーダル内のスクロール量が減り、イベント情報を俯瞰的に把握できる。

### 🟢 低優先度（将来）

#### 10. テーマカスタマイズUI
- **参考**: Linear
- **現状**: ダーク/ライトの2択のみ。
- **実装内容**: ベースカラー・アクセントカラーをユーザーが選択できる設定画面。LCHベースのカラー生成ロジックをJSで実装。
- **実装後の変化**: 企業ブランドカラーに合わせたカスタマイズが可能になり、他部署への展開時に対応しやすくなる。

#### 11. キーボードショートカット一覧（`?` キー）
- **参考**: Linear、Raycast
- **現状**: `⌘K` の `kbd` 表示は検索バーにあるが、他のショートカットは非公開。
- **実装内容**: `?` キーでモーダルを開き、利用可能なショートカット一覧をカテゴリ別に表示。
- **実装後の変化**: PC利用の編集者が習熟しやすくなり、操作効率が継続的に向上する。

#### 12. ドキュメント的なイベントノートエディタ
- **参考**: Craft.do、Notion
- **現状**: `note` フィールドがtextarea1本。
- **実装内容**: `/` コマンドでリスト・見出し・チェックリストを挿入できるブロックエディタ（Tiptap等）を導入。
- **実装後の変化**: イベントの詳細メモが構造化され、準備物リストとの連携も強化できる。

#### 13. カレンダー上部KPIバー
- **参考**: Vercel Dashboard、Fathom Analytics
- **現状**: カレンダーヘッダーは年月ナビゲーションのみ。
- **実装内容**: カレンダー上部に「今月 N件 / 準備完了率 XX% / 予算合計 ¥XXX,XXX」を表示するインジケーターバー。
- **実装後の変化**: カレンダー画面だけで月次サマリーを把握でき、別画面への遷移が減る。

## デザイン部門からのブリーフまとめ

### コアデザイン原則（調査から導出）

1. **情報のヒエラルキーを視覚化する**: Linear・Fathom Analytics に倣い、重要な数値は大きく、補足情報は小さく・薄く。現状は全体的にサイズ差が小さい。

2. **モーションは「伝える」ために使う**: Raycast・Linear のスプリングアニメーションのように、開閉・遷移の物理的フィードバックで状態変化を伝える。フェードだけでは「何が起きたか」が伝わりにくい。

3. **モバイルファーストのタップターゲット**: Cron の週ビューに倣い、セル・ボタンの最小サイズを 44×44px 以上に保つ。現場スタッフが手袋やグローブ越しに操作する可能性も考慮。

4. **カラーは「意味」で使う**: Vercel Dashboard に倣い、彩色は「状態を伝える」ためだけに使用。装飾的な色使いは最小化し、信号機ロジック（緑/黄/赤）を徹底する。

5. **余白は「呼吸」のために確保する**: Craft.do に倣い、コンテンツを詰め込まず、読み取りに必要な空白を意識的に設計する。
