## 1. アプリ概要（なぜ作るか）

### 背景・課題

EX事業部では、イベントで使用する備品・消耗品の在庫管理を現状スプレッドシートまたは口頭で運用している。
イベント管理アプリ（本アプリ）によってイベントのスケジュールと準備物管理はWebアプリ化できたが、
**定常的な在庫（倉庫にある物品の個数）の管理と、魚類などの生体在庫の管理は別物**であり、
現在のイベント単位の準備物リストでは対応しきれていない。

### 解決したいこと

| 現状の困りごと | 在庫管理アプリで解決 |
|------------|-----------------|
| 備品の残数が分からない（各自バラバラに把握） | アイテム別の個数をリアルタイムで共有 |
| 「これ追加購入する？」の検討がチャットや口頭で散らばる | アイテムに追加購入メモ・URLを直接紐付け |
| イベントで使う魚の種類・数が直前まで曖昧 | イベントごとの魚在庫を専用画面で管理 |
| 在庫に関する連絡が別のツールに散らばっている | アプリ内の簡易チャットで文脈を共有 |

### Why（企画部門の見解）

イベント管理と在庫管理は**別のライフサイクル**を持つ。イベントは「計画→実施→終了」だが、
在庫は「補充→使用→枯渇→補充」という循環サイクル。同一アプリに統合するより**独立したアプリとして作り、
ナビゲーションで行き来できる**設計が保守性・UX両面で優れている。

## 2. 主要機能リスト

オーナー指示から4つの主要領域に整理した。

### 機能A: 一般備品在庫管理

| # | 機能 | 説明 |
|---|------|------|
| A-1 | アイテム登録 | 備品名・カテゴリ・現在の個数を登録する |
| A-2 | 個数の増減 | `+1 / -1` ボタンまたは直接入力で個数を更新する |
| A-3 | 在庫アラート | 設定した下限個数を下回ったら警告表示する |
| A-4 | 追加購入メモ | アイテムに「検討中・要購入・購入済み」の購入ステータスを付ける |
| A-5 | 購入URL | Amazon・楽天等の購入先URLをアイテムに紐付け、タップで開ける |
| A-6 | 備考メモ | アイテムごとに自由記入メモを添付できる |

### 機能B: 魚在庫管理

| # | 機能 | 説明 |
|---|------|------|
| B-1 | 魚種登録 | 魚の種類名・個数・保管場所を登録する |
| B-2 | 個数の増減 | 入荷・放流・死亡等の理由と一緒に個数を更新できる |
| B-3 | イベント紐付け | 特定のイベント（イベント管理アプリのイベントID）に何種類・何匹使う予定かを設定する |
| B-4 | 必要数チェック | イベント日が近づいたときに在庫数と必要数を比較して不足を表示する |

### 機能C: 追加購入検討

| # | 機能 | 説明 |
|---|------|------|
| C-1 | 購入検討リスト | 「要検討」ステータスのアイテムを一覧表示する専用ビュー |
| C-2 | URLプレビュー | 購入URLのファビコン+ドメインをサムネイル風に表示する |
| C-3 | 購入完了マーク | 購入が完了したらワンタップでステータスを「購入済み」に変える |

### 機能D: 簡易チャット

| # | 機能 | 説明 |
|---|------|------|
| D-1 | グローバルチャット | アプリ全体の連絡用チャットルーム（1チャンネル） |
| D-2 | アイテムコメント | 特定の備品・魚種に紐付いたスレッド形式のコメント |
| D-3 | 送信者表示 | Googleアカウント名・アバターを表示（本アプリと同じ認証） |
| D-4 | 既読管理（任意） | 未読数バッジ（優先度低・後回し可） |

## 4. イベント管理アプリとの連携方法（ナビゲーション案）

### 基本方針

2つのアプリは**独立したVercelプロジェクト**として管理し、URLで行き来する。
同一Firebaseプロジェクト（または専用プロジェクト）を共有することでデータ連携を実現する。

### 遷移方法の選択肢

| 案 | 方法 | メリット | デメリット |
|----|------|---------|---------|
| 案1 | ヘッダーにリンクボタン設置 | 実装が最もシンプル | URLが変わるのでページリロードが発生 |
| 案2 | サイドバーに「在庫管理」メニュー追加 | 本アプリのUIに自然に溶け込む | 本アプリのコードを修正する必要がある |
| 案3 | モバイル：ボトムナビに追加タブ | モバイルで最もスムーズ | ボトムナビの改修が必要 |

**企画部門推奨: 案2（サイドバーリンク）+ 案3（モバイルボトムナビ）の組み合わせ**

### 実装イメージ（本アプリ側の変更）

現在の `App.tsx` のサイドバーに以下を追加：

```tsx
// サイドバー下部に「アプリ切替」セクションを追加
<div className="mt-auto p-4 border-t border-slate-100">
  <a
    href={INVENTORY_APP_URL}
    className="flex items-center gap-3 px-3 py-2 rounded-lg text-slate-600 hover:bg-slate-100"
  >
    <Package size={14} />
    <span className="text-xs font-bold">在庫管理アプリ</span>
  </a>
</div>
```

モバイルのボトムナビ（現在は `calendar / prep / archive` の3タブ）に「在庫」タブを追加、
または外部リンクアイコンとして区別表示する。

### 在庫管理アプリ側（本アプリへの戻り方）

在庫管理アプリのヘッダーに「イベント管理へ」ボタンを常設する。

## 5. データ設計案（Firestoreコレクション構造）

### 共有データ

本アプリと在庫管理アプリで**同一Firebaseプロジェクトを使う**ことを前提とする。
これにより `events` コレクションを在庫管理アプリからも参照できる。

### 新規コレクション

```
Firestore
├── events/                         ← 既存（本アプリ）
├── staff/                          ← 既存（本アプリ）
│
├── inventoryItems/                 ← 【新規】一般備品
│   └── {itemId}/
│       ├── name: string            # アイテム名
│       ├── category: string        # カテゴリ（消耗品/道具/備品 etc）
│       ├── quantity: number        # 現在個数
│       ├── minQuantity: number     # 下限アラート個数
│       ├── purchaseStatus: 'none' | 'considering' | 'ordered' | 'purchased'
│       ├── purchaseUrl: string     # 購入先URL
│       ├── note: string            # 備考
│       ├── updatedByUid: string
│       ├── updatedAt: Timestamp
│       └── comments/              # ← サブコレクション（アイテムコメント）
│           └── {commentId}/
│               ├── text: string
│               ├── authorUid: string
│               ├── authorName: string
│               └── createdAt: Timestamp
│
├── fishInventory/                  ← 【新規】魚在庫
│   └── {fishId}/
│       ├── species: string         # 魚の種類名（例: マダイ、ブリ）
│       ├── quantity: number        # 現在個数
│       ├── location: string        # 保管場所・水槽名
│       ├── note: string
│       ├── updatedByUid: string
│       ├── updatedAt: Timestamp
│       └── history/               # ← サブコレクション（個数変更履歴）
│           └── {historyId}/
│               ├── delta: number  # 増減数（+なら入荷, -なら使用/死亡）
│               ├── reason: 'arrival' | 'usage' | 'death' | 'other'
│               ├── note: string
│               ├── authorUid: string
│               └── createdAt: Timestamp
│
├── fishEventAssignments/           ← 【新規】魚とイベントの紐付け
│   └── {assignmentId}/
│       ├── fishId: string          # fishInventory の ID
│       ├── eventId: string         # events の ID（本アプリ）
│       ├── plannedQuantity: number # 使用予定個数
│       └── note: string
│
└── inventoryChat/                  ← 【新規】チャット
    └── {messageId}/
        ├── text: string
        ├── authorUid: string
        ├── authorName: string
        ├── authorPhotoURL: string
        └── createdAt: Timestamp
```

### Firestore セキュリティルール方針

- 認証済みユーザー（`allowedUsers` コレクションに存在するUID）のみ読み書き可
- 本アプリと同一の `allowedUsers` コレクションを参照することで、ユーザー管理を一本化する

## 6. 未決事項・確認事項（オーナーへの質問）

以下は実装開始前にオーナーの意思決定が必要な項目。

### Q1. Firebaseプロジェクトの共有について

> 在庫管理アプリは本アプリと**同一のFirebaseプロジェクト**を使いますか？
> それとも別プロジェクトを新設しますか？

- **同一**: ユーザー管理・イベントデータを共有できる。コスト・管理コスト低。
- **別プロジェクト**: アプリ間を完全に分離できる。将来的な分離が楽。

**企画部門推奨**: 同一プロジェクト共有（現実的に管理しやすい）

---

### Q2. Vercelのデプロイについて

> 在庫管理アプリは**新しいVercelプロジェクト**として独立デプロイしますか？
> それとも本アプリ（ivent-schedule-ex）のルーティングに組み込みますか？

- **独立デプロイ**: URL例 → `inventory.ex-app.jp`
- **サブパスに組み込み**: URL例 → `ivent-schedule-ex.vercel.app/inventory`

**企画部門推奨**: 独立デプロイ（保守・デプロイのサイクルが分離できる）

---
